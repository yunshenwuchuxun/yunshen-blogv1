## 引言：两句话写一篇论文

在上一篇文章中，我们系统梳理了 [Agent Skills 开放标准](/blog/agent-skills)的全貌——从 Progressive Disclosure 的设计哲学，到 Claude Code / OpenAI Codex / LangChain 的跨平台实现对比。那篇文章回答的是"Agent Skills 是什么"和"为什么需要它"。

本文回答一个更具体的问题：<strong>一个真实的、产品级的 Agent Skill 长什么样？</strong>

`latex-arxiv-SKILL` 是一个基于 Agent Skills 标准构建的项目，它为 OpenAI Codex CLI 和 Claude Code 提供了一套端到端的 arXiv 综述论文写作工作流。用户只需输入两句话——第一句描述主题，第二句确认方向——Agent 就会自动完成文献检索、大纲规划、逐章节写作、引用验证、LaTeX 编译，最终交付一份可提交 arXiv 的双栏 PDF。

这不是一个玩具项目。它的示例产出是一篇关于 SOTA 生成式图像模型的完整综述论文，包含 8 个章节、100+ 条经过验证的 BibTeX 引用、6 种可视化类型，以及一份编译通过的 PDF。而驱动这一切的核心，是四个相互协作的 Agent Skills。

## 项目全景

项目的顶层结构清晰地体现了 Agent Skills 的"文件夹即 Skill"哲学：

```text
latex-arxiv-SKILL/
├─ .codex/skills/
│  ├─ arxiv-paper-writer/          # 核心 Skill：端到端论文工作流
│  ├─ latex-rhythm-refiner/        # 辅助 Skill：写作节奏润色
│  ├─ collaborating-with-claude/   # 协作 Skill：Claude Code 桥接
│  └─ collaborating-with-gemini/   # 协作 Skill：Gemini CLI 桥接
├─ AGENTS.md                       # 仓库级工作流硬约束
├─ agent-skills-standard.md        # Skill 编写规范文档
├─ example/                        # Agent 生成的示例论文工程
└─ .claude                         # Claude Code 兼容入口
```

四个 Skills 各司其职：

| Skill | 职责 | 类型 |
|-------|------|------|
| arxiv-paper-writer | 端到端综述论文工作流（检索→规划→写作→编译） | 核心工作流 |
| latex-rhythm-refiner | 对已完成的 LaTeX 正文做句段节奏优化 | 后处理 |
| collaborating-with-claude | 通过桥接脚本调用 Claude Code CLI 获取第二意见 | 跨模型协作 |
| collaborating-with-gemini | 通过桥接脚本调用 Gemini CLI 获取第二意见 | 跨模型协作 |

这种"一个核心 Skill + 多个辅助 Skill"的组合模式，在 Agent Skills 生态中被称为 <strong>Skill Bundle</strong>——多个 Skills 共同服务于一个复杂领域任务，各自保持独立但可被编排调用。

## 核心架构：Issues-Driven 写作工作流

`arxiv-paper-writer` 的设计灵感来自软件工程中的 Issues-Driven Development：把"写论文"拆解成一系列可跟踪、可验证的工序（Issues），用 CSV 文件作为执行合同（Contract），用门禁（Gate）控制推进节奏。

![学术论文写作工作流](/static/blog/latex-arxiv-skill/workflow.svg)

整个工作流分为六个阶段：

### 阶段 1：Kickoff（框架期）

Agent 接收用户的主题描述后，执行以下动作：

1. 将主题翻译为检索关键词，做一轮 10-20 篇的轻量文献扫描
2. 提出 2-4 个候选标题
3. 通过脚手架脚本创建 LaTeX 项目骨架和 Plan 文件

```bash
python3 scripts/bootstrap_ieee_review_paper.py --stage kickoff --topic "<topic>"
```

这一步的关键输出是 `plan/<timestamp>-<slug>.md`——一份包含大纲、可视化计划和澄清问题的计划文件。此时 `main.tex` 中只有章节标题和要点占位，<strong>严格禁止写入正文</strong>。

### 阶段 2：Gate（审批期）

这是整个工作流最关键的设计决策：<strong>在用户明确批准之前，Agent 不得写入任何正文</strong>。

Plan 文件中包含一个显式的门禁复选框：

```markdown
## Kickoff Gate (must be confirmed before writing)
- **STOP**: Do not write prose into `main.tex` until this gate is confirmed.
- [ ] User confirmed scope + outline in chat
```

只有当这个复选框被勾选为 `[x]` 后，后续的 Issues CSV 生成脚本才会执行。脚本会主动检查这个勾选状态——这不是靠 Prompt 约束，而是靠代码强制执行的硬门禁。

### 阶段 3：Issues（执行合同）

门禁通过后，Agent 生成 Issues CSV——一份结构化的任务清单，每行对应一个可交付的工作单元：

```text
ID,Phase,Title,Target_Citations,Visualization,Status,Verified_Citations
R1,Research,Skeleton main.tex (no prose),0,N/A,TODO,0
W1,Writing,Introduction and scope,8,None,TODO,0
W2,Writing,Foundations and building blocks,10,Fig 1 taxonomy,TODO,0
...
Q1,QA,Citations: 100% verified,0,N/A,TODO,0
Q2,QA,Compile cleanly (latexmk),0,N/A,TODO,0
```

Issues CSV 是工作流的执行合同。它有严格的 schema 校验（通过 `validate_paper_issues.py` 脚本），状态只能按 `TODO → DOING → DONE` 单向推进。当写作过程中发现新任务时，Agent 必须向 CSV 中插入新行（如 `W6a`、`Q5`），而不是做未跟踪的工作。

这种设计直接解决了 Agent 写作中最常见的问题：<strong>漫无边际和不可审计</strong>。

### 阶段 4：Writing Loop（按 Issue 迭代）

Agent 逐个 Issue 推进：

1. <strong>Research</strong>：为当前章节检索 8-12 篇相关论文
2. <strong>Write</strong>：撰写正文，遵循"不超过 3 句话无引用"的密度要求
3. <strong>Verify</strong>：通过 Web 搜索验证每条引用的真实性
4. <strong>Update</strong>：将 Issue 标记为 DONE，更新 `Verified_Citations` 计数
5. <strong>Compile</strong>：每完成有意义的变更就编译一次，修复排版警告

当一个 Issue 在写作过程中膨胀（比如需要新增图表或子章节），Agent 必须拆分出新的 Issue 行，重新校验 CSV 后继续。

### 阶段 5：Refinement（润色）

所有写作 Issues 完成后，Agent 调用另一个 Skill——`latex-rhythm-refiner`——对正文做句段节奏优化。

### 阶段 6：QA + Compile（交付）

运行内部 QA 清单，编译 PDF，确保零编译错误和零未定义引用后交付。

## 设计模式：脚本确定性层

`arxiv-paper-writer` 最值得学习的设计模式是<strong>将确定性行为封装到脚本中，只让 Agent 处理非确定性的创造工作</strong>。

### 为什么需要脚本

直觉告诉我们：既然 Agent 能写代码，为什么不让它直接操作文件系统？答案是<strong>可复现性和安全性</strong>。

考虑"创建 LaTeX 项目骨架"这个任务。如果交给 Agent 自由发挥，每次执行可能产生不同的目录结构、不同的模板内容、甚至不同的文件编码。而 `bootstrap_ieee_review_paper.py` 脚本保证了每次执行的结果完全一致——相同的模板、相同的目录、相同的命名约定。

### 五个脚本的职责划分

```text
scripts/
├─ bootstrap_ieee_review_paper.py   # 脚手架：一键创建项目 + Plan / Issues
├─ create_paper_plan.py             # 计划生成：Plan 文件 + Issues CSV
├─ validate_paper_issues.py         # 校验器：Issues CSV schema 强制校验
├─ compile_paper.py                 # 编译器：LaTeX 编译 + 页数统计
├─ arxiv_registry.py                # 注册表：arXiv 检索 + BibTeX 缓存
└─ paper_utils.py                   # 工具集：共享的字符串/路径/格式函数
```

每个脚本都遵循相同的设计原则：

- <strong>标准库实现</strong>：全部使用 Python 标准库，不依赖第三方包
- <strong>CLI 入口</strong>：通过 `argparse` 提供 `--help`，Agent 先读帮助再调用
- <strong>幂等性</strong>：重复执行不会产生副作用
- <strong>失败响亮</strong>：校验失败时输出明确的错误信息，不静默跳过

这种"脚本做骨架，Agent 做血肉"的分工，在 Agent Skills 标准文档中被称为 <strong>Black-box Helper Pattern</strong>。SKILL.md 中甚至明确写道：

> Always run scripts with `--help` first. DO NOT read the source until you try running the script first... They exist to be called directly as black-box scripts rather than ingested into your context window.

核心理念是保护 Agent 的上下文窗口——脚本越大越不应该被读入上下文。

## arXiv Registry：可复现的文献检索

`arxiv_registry.py` 是整个项目中最精巧的组件。它用一个本地 SQLite 数据库，把"在 arXiv 上搜索论文"这个天然不可复现的操作变成了可审计、可缓存、可导出的确定性流程。

### 数据模型

```text
schema_meta          → schema 版本元信息
works                → 以 base arXiv ID（去除 vN 后缀）为主键的论文元数据
authors / work_authors → 作者去重 + 顺序关联
searches / search_results → 检索查询与结果快照（含原始 XML 和哈希）
bibtex               → 每篇论文最新一次抓取的 BibTeX（含 SHA-256）
fetches              → 远端抓取日志（kind/url/status/hash/bytes）
citation_keys        → 为每篇论文生成并固化的 citation key
```

### 核心工作流

```bash
# 1. 初始化注册表
python3 scripts/arxiv_registry.py --project-dir <paper_dir> init

# 2. 执行检索并持久化结果
python3 scripts/arxiv_registry.py --project-dir <paper_dir> search \
  --query "diffusion transformer image generation" --max-results 20

# 3. 抓取并缓存 BibTeX
python3 scripts/arxiv_registry.py --project-dir <paper_dir> fetch-bibtex <arxiv_id>

# 4. 以稳定 citation key 导出到 ref.bib
python3 scripts/arxiv_registry.py --project-dir <paper_dir> export-bibtex <arxiv_id> \
  --out-bib <paper_dir>/ref.bib
```

<strong>Citation key 生成策略</strong>特别值得注意：取第一作者姓 + 年份 + 标题首词，规范化拼接为 `base_key`；若冲突则追加 arXiv ID 数字后缀。生成后写入 `citation_keys` 表，后续导出时复用同一 key，避免 citation key 在不同导出批次间漂移。

这里的设计哲学是：<strong>每一步远端交互都留下审计痕迹</strong>。搜索结果的原始 XML、每次 fetch 的 HTTP 状态码和响应哈希、BibTeX 的 SHA-256——全部持久化到 SQLite。当论文审稿人质疑某条引用的来源时，作者可以从注册表中精确回溯"这条 BibTeX 是什么时候、从哪个 URL、在什么搜索查询下获取的"。

## 跨模型协作：Claude / Gemini 桥接

项目中最具创新性的设计是两个协作 Skill——`collaborating-with-claude` 和 `collaborating-with-gemini`。它们让 OpenAI Codex（作为主 Agent）能够在写作过程中调用 Claude 或 Gemini 作为"第二意见"。

### 桥接架构

```text
Codex (主 Agent)
  │
  ├─ claude_bridge.py ──→ Claude Code CLI ──→ 结构化 JSON 返回
  │     ↕ SESSION_ID（多轮对话）
  │
  └─ gemini_bridge.py ──→ Gemini CLI ──→ 结构化 JSON 返回
        ↕ SESSION_ID（多轮对话）
```

桥接脚本的职责是将不同 CLI 工具的输出统一封装为一致的 JSON 结构：

```json
&#123;
  "success": true,
  "SESSION_ID": "session-abc123",
  "agent_messages": "Claude/Gemini 的回复内容",
  "error": null,
  "all_messages": []
&#125;
```

### 核心约束

这些协作 Skill 的 SKILL.md 中定义了严格的安全边界：

- <strong>不直接调用 CLI</strong>：必须通过桥接脚本，确保输出/会话处理一致
- <strong>仅请求 Unified Diff</strong>：对于代码变更，只允许请求 diff 格式，禁止直接修改文件
- <strong>文件引用优先</strong>：传递文件路径和行号，避免将大段代码粘贴到 prompt 中（节省 token）
- <strong>会话状态胶囊</strong>：每次协作后更新一个简短的状态摘要，保持上下文连贯

```text
[Claude Collaboration Capsule]
Goal: Review diffusion section for citation gaps
Claude SESSION_ID: session-abc123
Files/lines handed off: main.tex:120-180
Last ask: Check citation density in Section 3
Claude summary: Found 2 paragraphs without citations
Next ask: Propose specific papers to cite
```

这种"主 Agent 协调 + 辅助模型提供专业意见"的模式，本质上是 Anthropic 所说的 <strong>Orchestrator-Workers Workflow</strong> 的跨模型实现——Codex 是编排者，Claude/Gemini 是工人。

## 润色 Skill：纯约束驱动

`latex-rhythm-refiner` 是四个 Skills 中最简洁的一个——它没有脚本、没有模板、没有引用文档，<strong>整个 Skill 只有一个 SKILL.md，全部是约束和流程</strong>。

它的核心规则：

1. <strong>不增删、不移动任何 `\cite&#123;...&#125;`</strong>——引用是论文的骨架，不可触碰
2. <strong>句段长短交替</strong>——短句（5-12 词）、中句（13-22 词）、长句（23-35 词）混合使用
3. <strong>段落长度变化</strong>——短段（2-3 句）、中段（4-5 句）、长段（6-8 句）交替
4. <strong>去除填充词</strong>——"in order to" → "to"、"due to the fact that" → "because"
5. <strong>最小化过渡词</strong>——当结构已经暗示了逻辑关系时，删除 "However," / "Therefore,"

这个 Skill 的设计体现了一个重要的理念：<strong>并非所有 Skill 都需要脚本</strong>。对于"有明确规则但执行依赖语言理解"的任务，纯约束的 SKILL.md 就是最好的载体。脚本无法判断一段文字是否"节奏单调"，但 LLM 可以。

## 设计模式总结

剖析完整个项目后，可以提炼出以下 Agent Skills 设计模式：

### 模式 1：Gate-Contract-Verify 三段式

| 阶段 | 机制 | 实现方式 |
|------|------|---------|
| Gate（门禁） | 阻止 Agent 过早行动 | Plan 文件中的复选框 + 脚本检查 |
| Contract（合同） | 定义可验证的工作单元 | Issues CSV + schema 校验 |
| Verify（验证） | 确认每个单元符合验收标准 | 每个 Issue 的 Acceptance 字段 + QA 清单 |

这个模式适用于任何需要"分阶段交付、人类审批、可审计进度"的 Agent 任务——不只是论文写作，代码重构、数据迁移、文档生成都可以套用。

### 模式 2：Scripts for Determinism, Prompts for Creativity

```text
确定性任务（脚本处理）          非确定性任务（Agent 处理）
├─ 创建目录结构                 ├─ 决定论文大纲
├─ 复制模板文件                 ├─ 撰写正文
├─ 校验 CSV schema              ├─ 选择引用
├─ 编译 LaTeX                   ├─ 生成图表描述
├─ 导出 BibTeX                  └─ 润色句段节奏
└─ 解析 arXiv API 响应
```

判断标准很简单：<strong>如果一个操作每次应该产生完全相同的结果，放到脚本里；如果需要判断和创造力，留给 Agent</strong>。

### 模式 3：Skill Bundle 协作

```text
                 arxiv-paper-writer (核心)
                     │
         ┌───────────┼───────────┐
         │           │           │
  collaborating   collaborating  latex-rhythm
  -with-claude    -with-gemini   -refiner
  (第二意见)       (第二意见)      (后处理)
```

多个 Skills 可以组成一个 Bundle，在同一个工作流中被编排调用。核心 Skill 定义主流程，辅助 Skills 在特定阶段被触发。这种模式的关键是<strong>每个 Skill 保持独立可用</strong>——`latex-rhythm-refiner` 可以单独用于任何 LaTeX 项目的润色，不依赖 `arxiv-paper-writer`。

### 模式 4：Progressive Disclosure 的极致实践

`arxiv-paper-writer` 的 SKILL.md 是一个教科书级的 Progressive Disclosure 示例：

- <strong>Level 1（Metadata）</strong>：`name` + `description` 告诉 Agent"这个 Skill 用于写 arXiv 综述论文"
- <strong>Level 2（Instructions）</strong>：SKILL.md 正文包含完整的工作流、门禁规则和安全护栏
- <strong>Level 3（Resources）</strong>：7 个 references 文件提供深度指南（研究工作流、写作风格、引用规范、可视化模板、BibTeX 规则、模板用法、QA 清单），只在 Agent 执行到对应步骤时按需加载

SKILL.md 正文中明确标注了每个 reference 的加载时机：

> Open reference files only when a step calls them out.

这让 Agent 在正常写作时只加载核心指令（约 4000 tokens），仅在需要深度指导时才读取特定的 reference 文件。

每个示例都包含完整的产物链：

```text
example/<paper>/
├─ main.tex          # LaTeX 源文件
├─ ref.bib           # 验证过的 BibTeX 条目

```

## 思考：当 Agent 成为学术写作者

`latex-arxiv-SKILL` 项目引出了一些值得深思的问题：

<strong>Agent Skills 的边界在哪里？</strong> 这个项目证明了 Agent Skills 可以驱动远超"代码审查"或"测试生成"的复杂工作流。一篇综述论文涉及文献检索、学术写作、引用验证、LaTeX 排版、可视化设计——这些领域知识被编码在 SKILL.md 和 reference 文件中，让一个通用的 LLM 获得了领域专家的能力。

<strong>"执行合同"模式的可推广性</strong>。Issues CSV 本质上是一种结构化的 Agent 输出约束。不同于自由文本的 Prompt，CSV 的 schema 是可机器校验的——Agent 不可能"忘记"更新进度或跳过验证步骤，因为校验脚本会拦截。这种"把 Agent 约束编码为可验证的数据结构"的思路，比"在 Prompt 中反复强调规则"可靠得多。

<strong>跨模型协作的意义</strong>。`collaborating-with-claude` 和 `collaborating-with-gemini` 代表了一种新的 Agent 协作范式——不是人类监督 Agent，而是 Agent 之间互相审查。当 Codex 对某个章节的引用密度不确定时，它可以调用 Claude 做第二轮检查。这种跨模型的"同行评审"可能比单一模型的自我校验更可靠。
