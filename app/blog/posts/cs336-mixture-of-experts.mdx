当你想让一个语言模型"更强"，最朴素的办法是把它做大。但稠密（Dense）模型有个绕不开的代价：<strong>每多一份参数，每个 token 的前向计算就多花一份算力</strong>——参数量和计算量被死死绑在一起。于是出现一个尴尬的局面：我们明明知道"参数越多通常越强"，却被推理成本卡住了脖子。

混合专家模型（Mixture of Experts, MoE）就是为了拆开这个绑定而生的。它的核心主张可以用一句话概括：<strong>容量大，但计算稀疏</strong>——模型整体可以拥有海量参数，但每个 token 只激活其中很小一部分。今天的万亿参数大模型（DeepSeek-V3、Kimi-K2 等）几乎都建立在这个思想之上。

本文对应斯坦福 CS336 第四讲，也对应 Datawhale diy-llm 第五章。原文内容详尽但略显冗长，这里按"<strong>动机 → 核心机制 → 路由方向 → 负载均衡 → 主流变体 → 工程挑战</strong>"的逻辑重新组织，核心概念配以重绘的示意图，并保留原文中有价值的论文架构图与实验图作为佐证。

## 一、动机：为什么要把 FFN 拆成专家

标准 Transformer 层由自注意力和前馈网络（FFN）两部分组成，其中 <strong>FFN 通常占据了模型大部分参数</strong>。在稠密模型里，无论输入是什么 token，都要完整地流经同一个 FFN——参数全部参与计算。

MoE 的做法是：把这个单一的大 FFN，替换成一组并行的"专家"（每个专家就是一个小 FFN），再加一个<strong>路由器（Router）</strong>来决定每个 token 该交给哪几个专家处理。

![从稠密 FFN 到 MoE 层：路由器为每个 token 只挑选 K 个专家](/static/blog/cs336-mixture-of-experts/moe-overview.svg)

这样一来，参数与算力就解耦了：你可以把专家数量堆到很多（总参数极大），但每个 token 只过其中 $K$ 个（实际计算量很小）。比如一个有 64 个专家、每次激活 8 个的 MoE 层，参数规模接近 64 个 FFN，但单 token 的算力只相当于 8 个 FFN。

## 二、核心机制：通用门控公式

MoE 层的计算可以用一个统一的门控（gating）公式描述：

$$
y = \sum_{i \in \mathcal{T}} G_i(x)\, E_i(x)
$$

其中 $E_i(x)$ 是第 $i$ 个专家的输出，$G_i(x)$ 是路由器给该专家分配的权重，$\mathcal{T}$ 是被 Top-K 机制选中的专家索引集合。<strong>稀疏性的关键就在 $\mathcal{T}$</strong>——它只包含少数几个专家，未被选中的专家对该 token 的贡献为零，根本不需要计算。

路由器本身很简单，通常就是一个线性打分层：

$$
h(x) = x \cdot W_g
$$

$W_g$ 是可学习的投影矩阵，把 token 表示映射成"对每个专家的打分"，再经 softmax 归一化、取 Top-K。下面是一个最小专家与路由的实现：

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    """一个专家就是一个标准的两层 FFN。"""
    def __init__(self, dim):
        super().__init__()
        self.ffn = nn.Sequential(
            nn.Linear(dim, dim * 4),
            nn.ReLU(),
            nn.Linear(dim * 4, dim),
        )

    def forward(self, x):
        return self.ffn(x)
```

接下来真正的设计空间，全在"<strong>路由器怎么选专家</strong>"这件事上。

## 三、三种路由方向：谁来选谁？

路由的本质是一个分配问题，而分配可以从两个方向发起——是 token 主动挑专家，还是专家主动挑 token？这两个方向（再加一个不需要训练的哈希路由）构成了三种主流路由方式。

![TC 与 EC 两种路由方向的对比：方向相反，负载特性与失败模式截然不同](/static/blog/cs336-mixture-of-experts/routing-tc-ec.svg)

### TC 模式：Token 选择专家（"学生找导师"）

每个 token 根据路由打分，主动选出分数最高的 Top-K 个专家。这是最直觉、也最常用的方式。

<strong>优点</strong>是语义完整——每个 token 都会被分配到它"最想去"的专家。<strong>致命缺点</strong>是<strong>负载不均衡</strong>：训练中路由器很容易形成"马太效应"，少数热门专家被绝大多数 token 选中而过载，其余专家长期闲置得不到训练，模型逐渐"偏科"。

```python
class TC_MoE(nn.Module):
    """Token Choice：每个 token 选 Top-K 个专家。"""
    def __init__(self, dim, num_experts, k):
        super().__init__()
        self.k = k
        self.router = nn.Linear(dim, num_experts)
        self.experts = nn.ModuleList([Expert(dim) for _ in range(num_experts)])

    def forward(self, x):  # x: [B, D]
        gate = F.softmax(self.router(x), dim=-1)
        topk_scores, topk_idx = gate.topk(self.k, dim=-1)   # 每个 token 自己挑
        out = torch.zeros_like(x)
        for i in range(self.k):
            weight = topk_scores[:, i].unsqueeze(1)
            for e_id, expert in enumerate(self.experts):
                mask = (topk_idx[:, i] == e_id).float().unsqueeze(1)
                if mask.sum() == 0:
                    continue
                out += expert(x * mask) * weight
        return out
```

### EC 模式：专家选择 Token（"导师挑学生"）

把方向反过来：每个专家从所有 token 里，主动挑出它最想处理的 Top-K 个。

由于每个专家的处理量被 Top-K <strong>硬性约束</strong>，负载天然均衡，几乎不需要额外的均衡手段。但代价是另一端——<strong>有些 token 可能不被任何专家选中</strong>，这些 token 拿不到任何专家的处理，只能靠残差连接"裸奔"到下一层，造成语义信息缺失。

```python
class EC_MoE(nn.Module):
    """Expert Choice：每个专家选 Top-K 个 token，负载天然均衡。"""
    def __init__(self, dim, num_experts, k):
        super().__init__()
        self.k = k
        self.router = nn.Linear(dim, num_experts)
        self.experts = nn.ModuleList([Expert(dim) for _ in range(num_experts)])

    def forward(self, x):  # x: [B, D]
        gate = F.softmax(self.router(x), dim=-1)
        scores_T = gate.transpose(0, 1)                      # [E, B]：转置后由专家挑 token
        topk_scores, topk_idx = scores_T.topk(min(self.k, x.size(0)), dim=-1)
        dispatch = x.new_zeros((x.size(0), len(self.experts)))
        for e in range(len(self.experts)):
            for t_idx, s in zip(topk_idx[e].tolist(), topk_scores[e].tolist()):
                dispatch[t_idx, e] = s
        out = torch.zeros_like(x)
        for e_id, expert in enumerate(self.experts):
            w = dispatch[:, e_id].unsqueeze(1)
            if (w > 0).sum() == 0:
                continue
            out += expert(x * (w > 0).float()) * w
        return out
```

> 一组直观的对比实验（33 个 token、10 个专家）：TC 模式下专家负载是 `[13, 13, 16, 14, 9, 6, 20, 19, 18, 4]`，最忙和最闲相差 5 倍；EC 模式下每个专家恰好 2 个，完全均衡，但代价是若干 token 完全没被处理。<strong>负载均衡和语义完整，是一枚硬币的两面。</strong>

### 哈希路由：不训练的路由器

还有一条"反常识"的路线：路由器干脆不学习，用<strong>固定的哈希函数</strong>把 token 直接映射到专家。常见做法是局部敏感哈希（LSH）——用随机超平面对 token 投影取符号：

$$
h_i(x) = \left\lfloor \frac{a_i^\top x + b_i}{\varepsilon} \right\rfloor
$$

其中 $a_i$ 是随机向量、$b_i$ 是随机偏置、$\varepsilon$ 控制桶宽。

```python
class LSHRouter(nn.Module):
    """用随机超平面符号哈希把 token 映射到专家，无需训练。"""
    def __init__(self, d_model, num_experts, n_hashes=8):
        super().__init__()
        self.num_experts = num_experts
        self.register_buffer('random_vectors', torch.randn(n_hashes, d_model))
        self.n_hashes = n_hashes

    def forward(self, x):
        signs = (x @ self.random_vectors.T > 0).long()        # 随机超平面取符号
        powers = 1 << torch.arange(self.n_hashes, device=x.device)
        hashes = signs @ powers
        return hashes % self.num_experts                       # 落桶 → 专家 id
```

哈希路由<strong>天然负载均衡、计算成本极低、无需训练</strong>，但语义表达能力弱——它无法根据任务动态调整分配，本质上是把"学一个好路由"换成了"随机但均匀"。三种路由的取舍可以归纳为：

| 路由方式 | 是否可学习 | 负载均衡 | 语义表达 | 主要风险 |
|---|---|---|---|---|
| TC（Token 选专家） | 是 | 差，需额外约束 | 强 | 专家过载 / 偏科 |
| EC（专家选 Token） | 是 | 天然均衡 | 中 | Token 被丢弃 |
| Hash（哈希路由） | 否 | 天然均衡 | 弱 | 无法动态适配 |

## 四、负载均衡：让专家"雨露均沾"

TC 模式的负载失衡是 MoE 训练最核心的工程问题。业界发展出了几类互补的策略：

| 策略 | 核心思路 | 代表工作 |
|---|---|---|
| 辅助负载均衡损失 | 在 loss 中加正则项，惩罚分配不均 | Switch Transformer |
| 容量控制 + 溢出机制 | 设专家容量上限，超出转 fallback | 多数 MoE 系统 |
| 动态无辅助损失均衡 | 用 bias 动态调整路由分数，无需 aux-loss | Loss-Free Balancing |
| 共享 + 路由专家池 | 共享专家保底覆盖，路由专家做专精 | DeepSeekMoE |

### 辅助损失：把"均衡"写进训练目标

前面反复说 TC 模式会"偏科"，但<strong>到底怎么把"别偏科"这件事告诉模型？答案是改训练目标</strong>——在原本的语言建模损失之外，加一个惩罚"分配不均"的<strong>辅助损失（auxiliary loss）</strong>。这是 Switch Transformer 确立、至今最主流的做法。最终的训练损失变成两项之和：

$$
\mathcal{L} = \mathcal{L}_{\text{LM}} + \alpha\, \mathcal{L}_{\text{aux}}
$$

其中辅助损失定义为（$N$ 为专家数）：

$$
\mathcal{L}_{\text{aux}} = N \sum_{i=1}^{N} f_i\, P_i
$$

这里藏着一个精巧的设计，关键在两个量的"分工"：

- $f_i$ 是<strong>实际被分到专家 $i$ 的 token 比例</strong>（基于 Top-K 硬分配的计数）。它告诉你"谁过载了"，但因为是离散计数，<strong>不可导</strong>，没法直接拿来回传梯度。
- $P_i$ 是<strong>路由器给专家 $i$ 的平均 softmax 概率</strong>，连续、<strong>可导</strong>。

两者相乘再求和，妙处在于：<strong>梯度只能通过可导的 $P_i$ 回传</strong>。当某个专家 $f_i$ 很大（过载），损失就会把它对应的 $P_i$"拽小"——于是路由器学会<strong>降低</strong>把 token 送往过载专家的概率，分配被推回均匀。可以验证：完全均衡（$f_i = P_i = 1/N$）时 $\mathcal{L}_{\text{aux}}$ 取最小值 $1$，越不均衡值越大。

![辅助负载均衡损失：失衡时损失偏高，梯度通过可导的 P 把过载专家的概率压下去](/static/blog/cs336-mixture-of-experts/aux-loss.svg)

系数 $\alpha$ 控制"均衡"与"任务效果"的拉扯，通常取 $0.01$ 量级：太小压不住失衡，太大又会干扰语言建模主目标。实现只有几行：

```python
def load_balancing_loss(gate_probs, expert_idx, num_experts):
    # gate_probs: [N, E] softmax 后的路由概率（可导）
    # expert_idx: [N]   每个 token 实际分到的专家（硬分配，不可导）
    N = gate_probs.size(0)
    f = torch.bincount(expert_idx, minlength=num_experts).float() / N  # 各专家 token 占比
    P = gate_probs.mean(dim=0)                                         # 各专家平均概率
    return num_experts * torch.sum(f * P)                             # 均衡时取最小值 1
```

<strong>Router z-loss：顺手稳住数值。</strong> 除了均衡，Switch / ST-MoE 还常加第二个正则项——它惩罚路由 logits 的幅度，防止 softmax 之前的数值过大引发溢出和训练不稳：

$$
\mathcal{L}_{z} = \frac{1}{B}\sum_{b=1}^{B}\left(\log\sum_{j=1}^{N} e^{h_j^{(b)}}\right)^2
$$

```python
def router_z_loss(logits):              # logits: [N, E]，softmax 之前
    return (torch.logsumexp(logits, dim=-1) ** 2).mean()
```

于是 MoE 的完整训练目标，是三项的加权和：`loss = lm_loss + α * aux_loss + β * z_loss`（典型 $\alpha \approx 0.01$、$\beta \approx 0.001$）。

<strong>DeepSeekMoE 的两级均衡。</strong> 后文会提到，它把辅助损失按粒度拆成<strong>专家级</strong> $\mathcal{L}_{\text{ExpBal}}$ 与<strong>设备级</strong> $\mathcal{L}_{\text{DevBal}}$：前者沿用上面的 $f_i P_i$ 形式约束单个专家，后者把同一设备上的专家归为一组、约束<strong>跨设备</strong>的计算负载，避免某张卡被打爆。

<strong>无辅助损失均衡（Loss-Free）：另一条路。</strong> 辅助损失有个副作用——它和语言建模主目标"抢梯度"，本质上是牺牲一点效果换均衡。DeepSeek-V3 改用一招更干净的办法：<strong>不加损失项，改加偏置</strong>。给每个专家一个可调偏置 $b_i$，<strong>只在 Top-K 选择时</strong>用 $s_i + b_i$ 排序，而<strong>门控权重仍用原始的 $s_i$</strong>；再用一个简单控制器更新偏置——过载的专家调小 $b_i$、闲置的调大，下一步就少分/多分一些 token。整个过程不产生梯度、不干扰主损失，却同样把负载推向均衡。

### 容量上限与 Token Dropout

最常用的硬约束是给每个专家设一个"班级人数上限"：

$$
C_{\text{expert}} = \left\lceil \frac{N}{E} \times \text{capacity\_factor} \right\rceil
$$

$N$ 是 token 总数，$E$ 是专家数，$\text{capacity\_factor}$ 通常取 $1.25$。一旦某专家被分配的 token 超过容量，超出的部分就被丢弃（Token Dropout）——不执行该专家的 FFN，只通过残差连接传到下一层。

![专家容量与溢出机制：超出容量的 token 被丢弃，只走残差连接](/static/blog/cs336-mixture-of-experts/capacity-overflow.svg)

容量因子是一个典型的权衡旋钮：调大能减少丢弃、保住语义，但削弱了均衡效果、增加算力浪费；调小则相反。下面是一个带容量控制与<strong>噪声门控</strong>（训练时给路由打分加噪声，鼓励路由器探索不同专家组合）的 Top-1 MoE 层：

```python
class MoELayer(nn.Module):
    def __init__(self, d_model, d_ff, n_experts=4, k=1,
                 capacity_factor=1.25, noisy_gating=True):
        super().__init__()
        self.n_experts, self.k = n_experts, k
        self.capacity_factor = capacity_factor
        self.noisy_gating = noisy_gating
        self.w_gating = nn.Linear(d_model, n_experts, bias=False)
        if noisy_gating:
            self.w_noise = nn.Linear(d_model, n_experts, bias=False)
        self.experts = nn.ModuleList([
            nn.Sequential(nn.Linear(d_model, d_ff), nn.GELU(),
                          nn.Linear(d_ff, d_model))
            for _ in range(n_experts)
        ])

    def _noisy_logits(self, x):
        logits = self.w_gating(x)
        if self.noisy_gating and self.training:                # 训练时注入噪声促进探索
            logits = logits + torch.randn_like(logits) * torch.sigmoid(self.w_noise(x))
        return logits

    def forward(self, x):                                      # x: [B, T, D]
        B, T, D = x.shape
        x_flat = x.view(B * T, D)
        scores = F.softmax(self._noisy_logits(x_flat), dim=-1)
        top1 = torch.argmax(scores, dim=-1)
        combine_w = scores.gather(1, top1.unsqueeze(1))
        capacity = int((B * T / self.n_experts) * self.capacity_factor) + 1

        out = torch.zeros_like(x_flat)
        for e in range(self.n_experts):
            idx = torch.nonzero(top1 == e, as_tuple=False).squeeze(-1)
            if idx.numel() == 0:
                continue
            idx = idx[:capacity]                               # 超出容量的 token 被丢弃
            out[idx] = self.experts[e](x_flat[idx])
        return (out * combine_w).view(B, T, D)
```

把这一层接入 Transformer Block 时，通常<strong>只把部分层的 FFN 替换为 MoE</strong>（而非每层都换），在 `Self-Attention → MoE/FFN` 的残差结构里即插即用：

```python
class TransformerBlock(nn.Module):
    def __init__(self, d_model, nhead, d_ff, use_moe=False, moe_params=None):
        super().__init__()
        self.attn = SimpleSelfAttention(d_model, nhead)
        self.ln1, self.ln2 = nn.LayerNorm(d_model), nn.LayerNorm(d_model)
        self.use_moe = use_moe
        self.ffn = MoELayer(**moe_params) if use_moe else nn.Sequential(
            nn.Linear(d_model, d_ff), nn.GELU(), nn.Linear(d_ff, d_model))

    def forward(self, x, mask=None):
        x = x + self.attn(self.ln1(x), mask=mask)
        x = x + self.ffn(self.ln2(x))                          # MoE 与稠密 FFN 接口一致
        return x
```

## 五、主流变体：从 Switch 到 DeepSeek

### Switch Transformer：把 Top-K 推到极致的 Top-1

Switch Transformer 做了一个激进的简化：<strong>每个 token 只选 1 个专家</strong>（Top-1）。这把路由开销和通信量都压到最低。它的两个关键设计：

- <strong>超容量直接跳过</strong>：溢出的 token 不执行专家 FFN、也不产生梯度，只靠残差传递。
- <strong>Router Z-loss</strong>：对路由 logits 的幅度加惩罚项，抑制 softmax 数值过大带来的不稳定。

![Switch Transformer 结构：Self-Attention 之后接 Switching FFN 层，每个 token 路由到单个专家](/static/blog/cs336-mixture-of-experts/switch-transformer.png)

### DeepSpeed-MoE：让 MoE 在系统层面跑得动

DeepSpeed-MoE 主攻工程效率，三个方向同时发力：<strong>参数效率</strong>（PR-MoE 用"固定 MLP + 专家残差纠错"减少冗余）、<strong>蒸馏加速</strong>（MoS 分阶段蒸馏，模型缩小约 3.7 倍）、<strong>系统优化</strong>（灵活的并行组合把 All-to-All 通信复杂度从 $O(p)$ 降到 $O(p/L)$，关键内核延迟下降约 6 倍）。

### DeepSeekMoE：细粒度专家 + 共享专家

DeepSeekMoE 提出两个被后续广泛采用的核心策略：

<strong>细粒度专家分割</strong>：把 $N$ 个专家切成 $mN$ 个更小的专家（每个约为原来的 $1/m$），同时激活的小专家数也放大 $m$ 倍。这样在算力不变的前提下，<strong>专家组合的空间呈指数级增长</strong>——从 16 选 2 的 $C_{16}^{2}=120$ 种，暴涨到 64 选 8 的 $C_{64}^{8} \approx 44$ 亿种，组合的灵活性带来更强的专精能力。

<strong>共享专家隔离</strong>：保留少数几个<strong>始终激活</strong>的共享专家，专门承载通用知识，让路由专家可以专注于差异化的专精能力。经验上"共享专家 : 激活的路由专家"约为 $1:3$ 较优。

![DeepSeekMoE 的两步演进：在 Top-2 路由基础上做细粒度分割，再隔离出共享专家](/static/blog/cs336-mixture-of-experts/deepseekmoe-arch.png)

消融实验显示，在总参数与激活参数都相同的前提下，"细粒度分割 + 共享专家隔离"在多个基准上稳定优于朴素 MoE，尤其在 TriviaQA、NaturalQuestions 这类知识密集任务上提升明显：

![相同参数预算下，细粒度分割与共享专家隔离带来一致的性能提升](/static/blog/cs336-mixture-of-experts/param-activation-compare.png)

DeepSeekMoE 还把负载均衡拆成<strong>专家级</strong>（$\mathcal{L}_{\text{ExpBal}}$）和<strong>设备级</strong>（$\mathcal{L}_{\text{DevBal}}$，约束跨设备计算负载）两层损失，并采用混合精度策略——传输用低精度（FP8 省带宽）、关键计算用中等精度（BF16 保稳定）。

### DeepSeek-V4：前瞻路由与 SwiGLU 软截断

更新的工作针对超大规模 MoE 的<strong>训练稳定性</strong>做了两个有代表性的改进：

- <strong>前瞻路由（Lookahead Routing）</strong>：利用前一步（$t-\Delta t$）的路由参数，为第 $t$ 步数据<strong>提前缓存专家分配</strong>，并通过异步检测仅在出现 Loss Spike 时触发。代价约 20% 额外计算，但能有效消除损失尖峰。
- <strong>SwiGLU 软截断</strong>：把 SwiGLU 的线性分量钳制在 $[-10, 10]$、门控分量上限钳制为 $10$，抑制异常值、改善训练稳定性。

## 六、MoE vs Dense：什么时候值得用

在数据和算力充足的前提下，MoE 通常比同等激活算力的稠密模型<strong>收敛更快、效果更好</strong>。下图（来自相关实验）对比了两者的训练曲线：达到相同 HellaSwag 表现，MoE 大约能省下 3 倍的 FLOPs / tokens，或在墙钟时间上快约 2 倍。

![MoE 与 Dense 的训练对比：达到相同效果可省约 3× FLOPs，或快约 2×](/static/blog/cs336-mixture-of-experts/moe-vs-dense.png)

但 MoE 不是免费午餐，落地时要权衡：

- <strong>数据与算力门槛</strong>：从零训练 MoE 需要足够多的数据才能"喂饱"众多专家；数据不足时，稀疏激活反而难以充分训练每个专家。
- <strong>显存占用</strong>：虽然计算稀疏，但<strong>全部参数都要驻留显存</strong>，对部署的内存要求很高。
- <strong>工程复杂度</strong>：路由、负载均衡、跨设备 All-to-All 通信，都比稠密模型复杂得多。

一个折中路线是 <strong>Upcycling</strong>——把已训练好的稠密模型复制扩展成 MoE 再继续训练，省去从零开始的成本，收敛速度取决于初始化质量。

## 七、工程挑战：以 Kimi-K2 为例

当模型规模来到万亿级（Kimi-K2 总参数约 1.04T、激活参数约 32.6B），MoE 的工程挑战被放大到极致：<strong>路由不均衡</strong>（部分专家过载、部分闲置）、<strong>通信压力</strong>（跨 GPU、跨节点频繁交换数据）、<strong>并行布局复杂</strong>（张量/流水线/专家/序列并行的组合）、以及<strong>训练推理不一致</strong>。

对应的工程对策包括：混合并行设计降低通信开销、截断重要性采样修正防止少数专家过载、自适应并行调度器自动调整策略、以及用 LoRA + 强化学习微调把 GPU 成本压到约 10%。这些都说明：<strong>在超大规模下，MoE 的胜负手已经从"模型设计"转移到了"系统工程"。</strong>

## 八、总结

MoE 的核心思想可以浓缩为三句话：

- <strong>用稀疏路由把参数与算力解耦</strong>——总容量可以很大，单 token 计算可以很小。
- <strong>路由方向决定了失败模式</strong>——TC 重语义但易失衡，EC 重均衡但丢 token，哈希简单但不智能；负载均衡与语义完整始终在拉扯。
- <strong>负载均衡与训练稳定是工程主战场</strong>——从辅助损失、容量控制，到细粒度专家、共享专家、前瞻路由，演进主线始终围绕"让每个专家都被充分、均衡地训练"。

从 Switch Transformer 的极简 Top-1，到 DeepSeekMoE 的细粒度 + 共享专家，再到 DeepSeek-V4 的稳定性优化，MoE 已经成为现代大模型不可或缺的骨架。理解它，就理解了今天万亿参数模型"既大又快"背后的根本权衡。
